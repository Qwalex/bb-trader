# Баг: SL подтягивался ниже безубытка после TP1

## Симптомы

При включённом «Подтягивание SL с TP1, глубина 1» (`TP_SL_STEP_START=tp1`, `TP_SL_STEP_RANGE=1`) после исполнения TP1 позиция закрывалась в убыток. В логе виден шаг вида:

```
SL подтянут после TP: после TP1 → SL 1,675 → 1,7281 (безубыток)
```

При этом фактический вход был `1.849`, то есть `1.7281` — это не безубыток, а уровень ниже входа. Проблема воспроизводилась и на лимитных, и на рыночных ордерах.

Пример сделки:

- Вход: `1.849`
- SL: `1.7281`
- TP: `1.925` ✓, `2`, `2.1`, `2.2`, `2.35`, `2.5`, `2.7`, `3`
- Плечо 4×, объём 5 USDT
- 3 подряд закрытые в минус сделки с одним и тем же паттерном.

## Первопричина

Файл: `apps/api/src/modules/bybit/bybit.service.ts`, метод `stepStopLossIfTpFilled`.

После расчёта целевого SL (для первого шага лестницы — безубыток `avgEntry − tick`) код сразу проверял, что цель «по верной стороне» от `markPrice`, и при «неверной стороне» **молча переносил SL на `mark ∓ 1 тик`**:

```ts
if (direction === 'long' && newSl >= markRef) {
  newSl = this.snapPriceToTickNum(markRef - tickNum, tickSize);
}
```

Сценарий:

1. TP1 исполнился на `1.925`.
2. Цена откатилась к `~1.7282`.
3. Код считает `newSl = 1.849 − 0.0001 = 1.8489` (BE).
4. `1.8489 ≥ 1.7282` → срабатывает «зажим»: `newSl = snap(1.7282 − 0.0001) = 1.7281`.
5. Текущий SL в БД `1.675`, `1.7281 > 1.675` → `improves=true` → SL применён.
6. Результат: SL стоит **ниже** входа (`1.7281 < 1.849`). Позиция закрывается в убыток, хотя в логе помечено «безубыток».

Тот же «зажим» портил и шаг «на предыдущий TP» в случаях, когда цена откатывалась настолько глубоко, что `mark − tick` оказывался ниже BE.

## Фикс

`apps/api/src/modules/bybit/bybit.service.ts`, `stepStopLossIfTpFilled`:

1. Явный «пол безубытка» `beSl = snap(avgEntry ∓ tick)` — тик-выровненный, не плавает из-за субтиковой точности `avgPrice` на мульти-филле.
2. После зажима к `mark` добавлена проверка «пола»: если `newSl` оказался слабее `beSl` (для лонга ниже, для шорта выше), **шаг пропускается целиком**:
   - `setTradingStop` не вызывается, SL на бирже не трогается;
   - `tpSlStep` в БД не продвигается → на следующем poll-цикле шаг повторится, и как только цена вернётся за BE, SL встанет ровно в безубыток;
   - в логе появляется предупреждение `TP_SL_STEP: SL был бы слабее BE — пропуск` с форматированными (тик-выровненными) значениями `mark`, `breakeven`, `wouldBeSl`.
3. Для шага «по предыдущему TP» проверка `avgPrice` сделана best-effort: если биржа временно отдаёт `avgPrice=0/пусто` (редкая UTA-гонка сразу после fill), шаг по-прежнему применяется с целью = `sorted[idxTp]` (предыдущий TP) — пол просто не активен. Раньше эта же проверка жила только в BE-ветке, и регрессии там нет.
4. Зажим к `mark` оставлен: для цели «предыдущий TP» при умеренном откате `mark − tick` по-прежнему ≥ BE и даёт зафиксировать часть профита. Ниже BE код больше не опустит SL.

## Поведение после фикса

- TP1 исполнен, цена откатилась к `1.7282`:
  - `beSl = 1.8489`, зажим даёт `1.7281 < 1.8489` → **пропуск**, SL остаётся на прежнем уровне (в примере — `1.675`, т.е. исходный план сделки сохранён).
- Цена вернулась выше `1.8489` — следующий цикл выставит SL в безубыток.
- Цена не вернулась — позиция закроется по исходному SL, т.е. по заранее известному максимальному убытку, а не по «ложному безубытку».
- TP2 исполнился, цена между BE и TP1 — зажим к `mark − tick` по-прежнему работает и блокирует прибыль выше BE.

## Ручная проверка (testnet)

1. Открыть лонг с TP1 чуть выше входа.
2. Дождаться исполнения TP1 и симулировать откат ниже входа.
   - Ожидается: в `/logs` запись `TP_SL_STEP: SL был бы слабее BE — пропуск`, SL на бирже не меняется.
3. Вернуть цену выше входа.
   - Ожидается: следующий poll выставит SL ровно в безубыток.
4. Повторить для шага TP2 → TP1, в т.ч. при глубоком откате ниже BE (должен быть тот же пропуск).
